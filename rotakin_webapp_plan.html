01 — Executive Overview

🎯 What We're Building

A full-featured browser-based webapp that automates the Rotakin CCTV audit process defined by SANS 10222-5-1-4. The key leap from v2 is the introduction of an image ingestion pipeline: engineers drop camera screenshots or exported frames into a folder and the system automatically analyses them, extracts %R measurements, validates audit steps, and produces a professional compliance report — all without manual data entry.

Problem Being Solved
  • ▸ Engineers manually read %R values from monitor screens — slow and error-prone
  • ▸ 7-step audit procedure done on paper or in disconnected spreadsheets
  • ▸ Report writing is manual and inconsistent between engineers
  • ▸ No automated evidence trail linking images to test results
  • ▸ Facial recognition scoring (10-line test) is tedious to record and calculate
Solution Delivered
  • ▸ Drop a folder of camera images → auto-detect Rotakin targets → measure %R
  • ▸ AI-assisted step validation and anomaly flagging per camera
  • ▸ One-click SANS-compliant PDF report with embedded evidence images
  • ▸ Full audit trail with image annotations and engineer sign-off
  • ▸ Automatic facial recognition scoring from face card images
ℹ️
Standards Compliance

The webapp is purpose-built for SANS 10222-5-1-4 (South African National Standard for Electronic Security Systems — CCTV). The %R measurement system and 5-tier level classification (Identification / Recognition / Observation / Detection / Monitoring) plus the optional Facial 12% standard are all hard-coded into the engine with user-editable overrides for other standards (BS EN 50132-7, NPSA, custom).

04 — Core Technical System

🖼️ Image Ingestion & Analysis Pipeline

This is the most technically novel part of the webapp. All image processing runs client-side in the browser using the Canvas API and WebAssembly — no server upload required for the core analysis.

⚠️
Folder Input Requirement

Browsers support folder drop via webkitdirectory attribute and the DataTransfer API. Users drop an entire folder and the app reads all images inside it. This works in Chrome, Edge, and Firefox without any server component. File System Access API can also be used for persistent folder watching.

Processing Flow — Per Image

01
File Ingestion & Validation
Accept JPG/PNG/TIFF/BMP. Check file is valid image. Parse filename for camera reference pattern (e.g. CAM-01_static.jpg, front-door_smear.png). Queue for processing.
02
Image Normalisation
Decode to Canvas. Normalise brightness/contrast for consistent detection. Record original resolution and aspect ratio. Create thumbnail (200×150) for gallery display.
03
Rotakin Target Detection
Run edge detection (Canny-like filter via Canvas pixel data). Identify tall vertical rectangular region matching Rotakin silhouette proportions (~1:5 width:height ratio). Locate bounding box.
04
%R Calculation
Measure detected figure height in pixels. Divide by total frame height in pixels. Multiply by 100. This is the %R value. Compare against configured SANS standards to assign level. Store raw and rounded values.
05
Step-Specific Analysis
Based on image type tag: static → measure resolution scale bar. smear → detect motion blur magnitude. colour → sample colour bar regions and score accuracy. face → attempt face/number identification.
06
Confidence Scoring & Flag
Every automated measurement gets a 0–100% confidence score. Low-confidence results (below 70%) are flagged yellow for engineer review. Very low (below 40%) flagged red — manual entry required.
07
Annotation Render
Overlay bounding box, measurement ruler, confidence percentage, and level badge onto a copy of the image. Store annotated image as base64 for report embedding. Original image preserved separately.
08
Camera Record Population
Push all extracted values into the associated camera record. Pre-fill step result fields. Trigger UI refresh. Engineer sees the auto-populated form with images and can verify/correct any value.
✅
Naming Convention for Auto-Assignment

Images named [camera-ref]_[step-type].[ext] are automatically associated. Supported step types: static, smear, colour, face, snapshot. Example: CAM-03_smear.jpg → Camera 3, Step 5 (smear test). Unrecognised filenames go to an "Unassigned" tray for manual drag-assignment.

Technical Implementation Notes

Canvas-Based Processing
// Core %R measurement function measureRotakinR(imageData) { const edges = sobelEdgeDetect(imageData); const bbox = findRotakinBBox(edges); if (!bbox) return { r: null, conf: 0 }; const figH = bbox.bottom - bbox.top; const frameH = imageData.height; const rValue = (figH / frameH) * 100; return { r: rValue.toFixed(1), conf: bbox.confidence, bbox }; }
WebWorker Threading
Image processing runs in a WebWorker to prevent UI freezing during batch analysis of large image sets. Progress is communicated back to the main thread via postMessage events. Each image is processed sequentially with a progress bar and estimated time remaining.

A worker pool of 2–4 workers (based on navigator.hardwareConcurrency) allows parallel processing of independent images.

07 — AI Integration

🤖 AI Layer Design

Three distinct AI functions: (1) image analysis for automated measurement, (2) natural language report generation, (3) anomaly detection and remediation advice.

AI Function 1 — Image Analysis

When a vision-capable model is available (Claude, GPT-4o), the image is sent alongside the Canvas-based measurements as a verification layer. The AI confirms or disputes the automated detection, catches edge cases (partial occlusion, unusual lighting), and provides a natural language description of what it sees in the frame.

Vision model required Optional verification Fallback to Canvas-only
AI Function 2 — Report Writing

Structured audit data is serialised and sent to the AI with a template-specific system prompt. The AI generates the narrative sections: Executive Summary, Findings, Recommendations. The engineer can regenerate sections individually or the entire report.

Claude API (preferred) Ollama (local) 5 templates
AI Function 3 — Anomaly Detection & Advice

Automatically flags patterns that indicate problems beyond simple threshold failures:

  • ▸ Camera that barely meets standard likely to fail in poor weather — flag for note
  • ▸ Motion blur score high despite good lux → shutter speed issue recommendation
  • ▸ Focus degradation across multiple cameras from same manufacturer → firmware issue
  • ▸ Facial recognition score very low for high %R → lens distortion or monitor calibration issue

Remediation suggestions are specific and actionable:

  • ▸ "Reduce lens size from 6mm to 4mm to increase coverage and meet Recognition requirement"
  • ▸ "Move camera 2m closer to reduce target distance from 18m to 16m"
  • ▸ "Upgrade from 2MP to 4MP sensor to achieve Identification at current mounting position"
🔑
API Key Management

API keys are stored in the browser's IndexedDB (never sent to any server other than the configured AI endpoint). The app can optionally use a proxy URL so the key never appears in client-side code. Keys are masked in the settings UI after entry.

💾
Storage Size Warning

Base64 images inflate size by ~33%. A 30-camera audit with 4 images each at 1MB = ~160MB raw = ~215MB base64. IndexedDB supports up to 50–80% of available disk space but may be slow. Strategy: store full-res images externally (File System Access API with user-granted folder), only store thumbnails and annotated copies in IndexedDB. Provide clear storage usage indicator in settings.
